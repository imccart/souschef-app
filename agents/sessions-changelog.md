## Session 54 — meal_ids + e2e harness + order-page polish
- **`grocery_items.meal_ids`** introduced (was `trip_items.meal_ids` pre-Phase A) → `grocery-sync.md`.
- **`grocery_items.receipt_acknowledged`** separates "user has acted on this receipt match via the receipt page" from `checked` (which can be set by grocery-list pre-checking). Receipt page is the canonical reconciliation step. Receipt match writes `receipt_acknowledged=0`; `/receipt/resolve` (any action) sets it to 1. Receipt page filters matched/substituted sections by `!receipt_acknowledged`.
- **E2E test harness** at `frontend/e2e/` → `deploy-workflow.md`.
- **Order page prior selections — authoritative availability**: `/order/search` does a per-UPC Kroger lookup for any pref UPC not in today's top-12 results. Drops prefs Kroger doesn't acknowledge (foreign-chain UPCs from receipts, discontinued SKUs). Out-of-stock and wrong-fulfillment items dropped instead of shown disabled. Whole enrichment block try/except'd — failures degrade to empty prior selections, never crash.
- **Order page end-state**: product cards / prior selections render guarded by `activeItem &&` so once everything is resolved the end-state panel shows alone.

## Session 62 — phantom-row bug class + skip-flag pattern
- **Phantom-row bug class** (commits `b5adfd1`, `bcb0204`, `d44a63a`) → `grocery-sync.md`. The receipt-tagged + ordered rows invisible-to-meal-sync class.

- **`/order/search` cache-read no longer overwrites live in-stock** (commit `8ed49c2`): Cache-read at `api.py:1839` was clobbering the live search response's mode-correct `in_stock` with a potentially-stale cached value. - **Deleted dead `swap_meal_smart` endpoint + `SwapPrompt` UI** (commit `a367d29`, -320 / +13 lines). The endpoint was in `api.py:174-332`. Frontend `setSwapPrompt(...)` was only ever called with null in `PlanPage.jsx`, so `SwapPrompt` never rendered and both the `action='preview'` (random replacement) and `action='confirm'` (DELETE removable + add new ingredients) branches were unreachable. - **Fixed `buy-elsewhere round-trip` test** (commit `a367d29`): was asserting `/grocery/undo` clears `buy_elsewhere`. ## Session 70 — receipt fixtures + parser hardening
- **Receipt fixtures collection** (~111 grocery receipts + 200 restaurant fixtures) staged in gitignored `scratch/receipts/`. Sources: Wikimedia Commons (3 Walmart, 1 Target, 1 Trader Joe's, 1 Save Mart, 1 99¢ Only, 1 Family Dollar, 1 Smith's-as-Kroger-banner gas-only, 1 Alexander's, 1 Tesco UK PD, 1 Hofer AT CC0, 7 Aldi DE) + GitHub clones (`cphouser/receiptapp` = 71 real Cal Poly project receipts: 40 Safeway / 13 TJ / 7 Costco / 5 New Leaf / 4 Grocery Outlet — no LICENSE file, local parser-tuning fair use only, do not commit) + ExpressExpense (MIT, 200 restaurant) + iTunes Search API (12 chain apps × ~8 screenshots ≈ 100 marketing screens; ~4 actually receipts: `costco_03` in-warehouse digital receipt, `heb_16` curbside summary, `heb_go_02/03` cart + checkout). Realistic reframe captured in `scratch/receipts/README.md`: per-chain sample is too small (3-40) and unlabeled to "build our own parser" — value is regression fixtures + footer-count seed data for the existing Vision parser. **Big public-source gap**: Wikimedia returns zero hits for Kroger / Publix / Whole Foods / HEB / Wegmans / Sprouts / Albertsons; realistic path to closing is user's own receipts + consenting friends-beta participants. Reddit was sandbox-blocked over PII concerns (correctly) — manual collection is the right path for that source.
- **PDF parser weighted-qty bug fix** (commit `eb09b2e`). `_parse_kroger_structured` (`reconcile.py:344`) only matched integer-qty lines (`\d+ x \$`); weighted lines (`2.28 lbs x $17.99 each (approx.)`) failed the regex and the walker captured the *weight line* as the item name with `price=None`. Real-world cost on a real Kroger basket: 5 weighted items (Beef Strip Steak, Bananas, Cosmic Crisp, Deli Turkey, Chicken Thighs = $77.84 = ~30% of basket value) silently dropped — never matched grocery rows, fell into Extras with garbage names, total-spend metrics 30% under-reported. **Bug had been latent since the parser was written.** One-line fix: regex branch `[\d.]+\s*lbs?\s*x\s*\$[\d.]+` inserted before the price walker so the loop continues past the weight line and finds the standalone `$price` line + the real product name. Post-fix verification on real PDF: `sum(price) = $254.26 = exact pre-tax subtotal` (was $176.42 before). Validated against the Kroger PDF in `scratch/receipts/kroger/kroger_real_2026-05.pdf`.
- **PDF item-count cross-check** (commits `bc4518c`, `6703572`). Kroger PDFs print `Item Details   N Items` in the order-summary block. `parse_receipt_pdf` now returns `(items, footer_count: int | None)`; `_extract_kroger_item_count` regex extracts the integer. `_process_receipt` (`api.py:_process_receipt`) compares `footer_count` to `sum((ri.get("qty") or 1) for ri in receipt_items)` — the qty-multiplier sum, not the line count, since Kroger's "58 Items" is the qty sum (39 distinct UPC lines on the test receipt). Writes `item_count_footer / item_count_parsed / item_count_gap` (gap omitted if zero) to the upload response and emits `logger.warning("Receipt item-count gap (%s): footer=%d parsed=%d gap=%+d user=%s", ...)` on mismatch. **No user-facing UI** — original commit had a yellow banner with "try re-uploading" copy; user pushed back ("annoying", correctly), banner stripped in the next commit. ## Session 74 — receipt matching fix + receipt page as queue
- **`diff_grocery_list` if/elif blocked fallback strategies** (commit `af07b22`). User uploaded Kroger PDF receipt; "maple syrup" and "bacon" didn't match their corresponding grocery rows.
